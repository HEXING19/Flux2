# Flux - 基于大语言模型的智能安全运营平台 (XDR) 需求开发方案

## 一、项目概述与目标解析

### 1.1 项目核心定位与目标
Flux 旨在打造一个融合 AI 的企业级智能安全运营平台，主要解决传统安全运营中**操作繁琐、门槛高、需要跨系统交互**的痛点。
通过引入大语言模型（LLM）实现自然语言转 API 调用的能力，将安全运营人员的**“业务意图”自动化转化为“技术执行”**，大幅提升安全处置与运营的效率。

### 1.2 核心业务约束与要求
- **部署形态约束**：要求支持在客户环境**快速、极简部署**以便于进行 POC 验证。为了达到“极简”部署，架构应当避免重量级依赖（如复杂的微服务架构、重型中间件等），推荐使用轻量级容器化（Docker Compose）单体架构打包，或提供一键部署脚本。
- **扩展性要求**：需要支持多家主流 LLM（智谱AI、OpenAI、DeepSeek 及自定义端点），并且可以通过配置灵活切换。
- **安全性要求**：由于涉及 XDR API 的高权限操作（如 IP 封禁、修改事件状态），“高危操作”必须经过用户二次确认，确保 AI 决策的安全可控。

## 二、系统架构设计方案

基于轻量化部署与可扩展的要求，系统整体可分为以下核心层：

### 2.1 用户交互层 (Frontend + 落地页)
- **落地页与认证系统**：
  - 需求：基于现成的 Python 接口和联动码认证，设计带有科技感、符合安全平台调性的独立落地页。
  - 功能：支持用户通过获取/输入联动码快速登录使用。
- **智能对话工作台 (Chat UI)**：
  - 支持类似 ChatGPT 的对话界面。
  - 支持 Markdown 渲染、表格渲染（用于展示事件列表）、高亮显示（用于严重/高危事件）、图表渲染（用于日志统计趋势图）。
  - 支持多轮对话上下文的状态保持与展示。

### 2.2 核心业务逻辑层 (Backend)
- **LLM Gateway (模型网关层)**：
  - 维护各模型提供商（Zhipu, OpenAI, DeepSeek, Custom）的连接配置与 API Key。
  - 提供连通性测试接口。
  - 负责 LLM 请求的容错、失败自动重试、流式响应（Streaming）处理。
- **Agent 核心大脑层 (Skill 路由与意图识别)**：
  - 接收用户自然语言指令，结合当前多轮对话的 Context（上下文），交给 LLM 进行意图识别。
  - **参数提取引擎**：精准提取 IP、时间范围、事件等级等参数；当参数不足时生成追问话术（渐进式交互）。
  - **Function Calling 调度器**：将提取出的意图匹配到注册好的 Skills 集合中，执行对应的 API。
- **Security Action 层 (XDR API 联动层)**：
  - 封装对目标 XDR 系统的 API 请求（如：通过 UUID 获取外网处置实体、批量修改事件等）。
  - 处理微步在线等外部 API 集成。

## 三、核心功能模块详细设计

### 3.1 智能对话与多轮交互引擎 (Agent Logic)
- **上下文感知 (Context Awareness)**：引入会话记忆（Session Memory），若用户在第一轮说“查询过去24小时的高危事件”，得到一张列表后，在第二轮说“把序号3的事件相关的恶意IP封禁掉”，系统必须能够根据上一轮生成的事件集合，准确找到“序号3”对应的源点 IP。
- **渐进式交互逻辑**：
  - 当触发某项 Skill 且关键必填参数（如“设备名称”或“封禁时长”）缺失时，系统会中止 API 调用，并在对话框中以友好的形式向用户提问补全参数。
- **高危操作阻断与确认机制**：
  - 封禁外网 IP、批量忽略事件等关键操作被标记为 `high_risk=True`。
  - 触发这些 API 前，系统会在对话流中生成一个包含了操作细节的可交互卡片（Action Card），必须点击“确认执行”按钮后，后端方可继续调用 XDR API。

### 3.2 多 LLM 提供商管理策略
- 为不同的厂商提供一套标准适配接口（采用类似 OpenAI 类 API 的兼容格式处理，因为智谱与 DeepSeek 均支持此类接口）。
- 实现统一的管理界面，允许用户动态添加、修改厂商的 API Key、API Base URL 和默认使用的模型名称（如 `deepseek-chat`）。
- 具备断线重连、模型调用限速控制及超时重试机制。

### 3.3 核心技能组 (Skills System) 详细设计

紧密贴合 `docs/api-ref/` 提供的现成接口，Skill 系统的设计如下：

#### 3.3.1 安全事件管理核心 Skills
- **Skill 1: 查询安全事件 (`search_security_events`)**
  - **能力**：调用 `查询安全事件列表.txt` 接口。
  - **提取参数**：时间范围（转化为最近时间的相对时间戳/时间段）、事件等级、状态、名称、攻击类型、源/目的IP。
  - **展现形式**：分页表格，严重/高危标红高亮。
- **Skill 2: 事件详情查询 (`get_event_details`)**
  - **能力**：调用 `根据事件uuId获取举证信息.txt` + `根据事件uuId获取事件外网ip处置实体.txt` 接口。
  - **提取参数**：事件 UUID 或上一轮上下文中的事件序号。
  - **展现形式**：综合卡片，包含基础信息、时间线、外网IP实体关联信息和深度攻击举证。
- **Skill 3: 事件状态处置 (`update_event_status`)**
  - **能力**：调用 `批量修改安全事件处置接口.txt`。
  - **提取参数**：事件 UUID(s)、目标状态（待处置/处置中/已处置/已忽略）、处置备注。
  - **交互要求**：高危操作，需经过操作卡片二次确认。

#### 3.3.2 实体封禁管理 Skills
- **Skill 4: 查询封禁状态 (`check_block_status`)**
  - **能力**：调用 `查询封禁地址策略列表.txt`。
  - **提取参数**：被查询的 IP/域名/URL、封禁类型、设备名称（可选）。
  - **展现形式**：展示是否封禁及关联的策略/规则详情（规则ID、有效期）。
- **Skill 5: 执行实体封禁 (`execute_entity_block`)**
  - **能力**：调用 `新增网侧封禁地址策略.txt` 等。
  - **提取参数**：实体信息、目标设备名称、封禁类型（源/目的/双向/URL/DNS）、封禁时长（临时/永久）。
  - **交互要求**：极高危操作！若用户未指定设备和时长，需主动追问；执行前必须向用户展示拟执行清单及影响，强制确认。
- **Skill 6: 外部威胁情报联动 (`query_threat_intelligence`)**
  - **能力**：对接微步在线等外部 API。
  - **功能**：自动获取恶意 IP 的地理位置、历史攻击记录、情报信誉打分等。需支持在设置中心配置微步 API Key。

#### 3.3.3 日志与数据分析 Skills
- **Skill 7: 网络日志统计 (`analyze_network_logs`)**
  - **能力**：调用 `查询网络安全日志总数接口.txt`。
  - **提取参数**：时间范围（必须转换为准确的时间区间）、源/目的IP、协议、端口。
  - **展现形式**：生成总数汇总卡片，并在前端利用 ECharts 等库呈现趋势图和不同维度的饼状图（若 API 支持聚合查询维度）。

### 3.4 自动化场景编排与推送系统 (Playbooks)
- **定位**：解放双手的高级功能，针对定时、周期性的安全运营工作进行全自动闭环处理。
- **方案实现**：
  - **配置中心**：允许管理员配置自动化规则（例如：每天早上 9:00 处理过去 24h 的 `严重` 和 `高危` 事件）。
  - **执行引擎 (Cron/Celery)**：
    1. 自动触发搜索事件 API 获取当日列表。
    2. 后台调用 AI (LLM) 进行自动分析，生成每个事件的处置建议及受影响资产画像。
    3. 整理出《日度严重/高危事件处置报告》。
  - **通知与协同**：
    - 支持配置企业微信 Webhook Robot 链接。
    - 引擎将生成的报告以及“一键确认封禁”的审批链接推送到企业微信。企业微信群内的人员可查阅详情，进而辅助确认决策。

## 四、项目工程化与部署规划

### 4.1 技术栈选型建议
鉴于落地需轻量、快速构建的要求：
- **后端**：Python (FastAPI 框架) - 高性能、自带 OpenAPI 标准，极为适合做 LLM 和 API Proxy 与编排。支持快速集成 `langchain` / `auto-gpt` 等工具链。
- **前端**：Vue 3 或 React 18，配合 Tailwind CSS 或组件库（如 Element Plus / Ant Design），集成打磨一份带有暗黑科技感的主题皮肤，以适配安全平台调性。
- **数据库**：SQLite（默认/极简模式，适合无缝携带与一键运行） / 备选 PostgreSQL（针对有一定并发的测试环境）。

### 4.2 部署打包策略 (POC 友好型)
- **Docker Compose 一键部署**：
  - 提供 `docker-compose.yml` 集成 Frontend (Nginx 代理), Backend (FastAPI API server) 和本地 SQLite 文件挂载。
  - 执行 `docker-compose up -d` 即可在任一装有 Docker 的 Linux/Mac 机器上分钟级启动。
  - 交付物中仅需包含镜像包 (`.tar` 导出离线镜像供无外网环境安装) 和运行脚本 `install.sh`。

### 4.3 研发阶段划分

- **Phase 1: 核心通道与登录验证 (预计 1-2 周)**
  - 落地页和联动码登录系统设计开发（包含界面设计与现成 Py 脚本的集成）。
  - LLM 网关建设，智谱、DeepSeek 等模型的对接和配置联通性测试。
  - 核心会话页面 UI 搭建，打通与后端的聊天长连接/流式接口。

- **Phase 2: Skills 系统对接与多轮交互 (预计 2-3 周)**
  - 基于提示词工程（Prompt Engineering）与 Function Calling，实现参数提取和多轮对话状态保持（关联上一轮序号等）。
  - 对接 `docs/api-ref/` 中的核心 API（安全事件查询、状态更新、实体封禁等）。
  - 前端支持表格呈现和图表呈现，增加**高危操作拦截确认机制**。

- **Phase 3: 自动化扩展与优化验收 (预计 1.5 周)**
  - 研发自动化场景编排系统并打通企微 Webhook。
  - 接入微步在线等扩展威胁情报。
  - 系统健壮性测试与体验优化（如对话框体验，异常请求重试）。
  - 生成 Docker 打包工程，编写使用与极简部署指南。

---
> 总结：本方案通过设计多LLM接入层、增强型Skills网关与前端拦截机制，配合极简轻量化的工程化打包手段，完全满足了 Flux 以对话形式驱动XDR API、实现高危操作管控、企业微信自动化闭环及客户端极简部署的核心诉求。
