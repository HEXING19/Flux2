# Flux - XDR 智能安全运营平台

## 一、项目概述

### 1.1 项目名称
Flux - 基于大语言模型的智能安全运营平台（XDR Security Operations Platform）

### 1.2 项目定位
Flux 是一个融合人工智能技术的企业级安全运营平台，通过自然语言交互简化安全操作流程，实现从"业务意图"到"技术执行"的自动化转换，提升安全运营效率。

### 1.3 项目部署形态
需要快速而且方面的部署到客户侧的电脑上进行验证，所以打包程序要尽可能简单，目的就是可以在客户侧快速完成部署，完成验证

## 二、核心功能需求

### 2.1 AI对话助手系统

#### 2.1.1 自然语言交互
- **需求描述**：用户通过自然语言描述操作意图，通过LLM意图理解后匹配具体的skills（如资产添加skills）联动XDR API接口自动执行
- **功能点**：
  - 支持中文自然语言输入
  - 上下文感知对话（记住之前的对话内容）
  - 智能参数提取（IP地址、事件ID、设备名称等）
  - 渐进式交互（参数缺失时主动询问）
  - 高危操作需要用户二次确认
  - skills概述描述如2.1.3章节描述，详细描述见后面其他章节

#### 2.1.2 多LLM提供商支持
- **需求描述**：支持多家主流大语言模型服务
- **支持的提供商**：
  - 智谱AI
  - OpenAI
  - DeepSeek（deepseek-chat）
  - 自定义端点
- **功能点**：
  - 提供商连通性测试
  - 模型切换配置
  - API密钥管理
  - 请求失败自动重试

#### 2.1.3 技能 Skills 系统架构
- **需求描述**：提供预设的快捷操作技能，用户可通过自然语言触发。Skills 必须采用面向对象（OOP）的设计模式实现，每个 Skill 是一个独立且完备的领域专家类（Class），而非简单的无状态函数。对应的 API 接口文档详细定义在 `docs/api-ref` 文件夹中。
- **架构与设计要求**：
  1. **独立的技能基类（BaseSkill）**：需提供统一的执行接口（如 `execute`）、参数 Schema 定义抽象、并且需内置参数完整性校验机制。
  2. **渐进式追问防线**：当 LLM 提取的参数缺少当前 Skill 必须的传参时，Skill 不应直接引发运行时错误，而是主动抛出状态或自定义异常（如 `MissingParameterException`），由 Agent 底层拦截接管并向用户发起友好的自然语言追问。
  3. **实际 API 对接**：各个技能必须通过构建通用的 API Requester，严格依据 `docs/api-ref` 中的契约动态组装并向真实服务发起外部 HTTP 请求。
- **核心技能列表**：
  1. 查询安全事件 - 多维度筛选安全事件
  2. 查看IP实体 - 获取事件关联的外网IP实体
  3. 事件详细举证 - 展示攻击时间线和详细证据
  4. 更新事件状态 - 批量更新处置状态
  5. IP封禁管理 - 查询和执行IP封禁
  6. 日志统计分析 - 网络安全日志数量统计

#### 2.1.5 落地页以及认证登录
接口使用现成的py文件，使用联动码登录，你需要帮我设计落地页，你可以自己整合到代码仓库中

### 2.3 Skills 上下文状态与多轮关联要求
- **需求描述**：由于基于自然的交互，Skills 系统必须具备上下文短期记忆机制（通常设计为 `SkillContextManager`），确保对列表数据的索引进行状态寻址。
- **具体能力要求**：
  1. **参数与意图继承**：当前对话具备引用/继承上一轮对话实体参数的能力（如：用户刚查过一个 IP 的事件，下一轮直接说“封禁它”，系统应能顺延提取到该 IP）。
  2. **列表索引映射管理（关键要求）**：当某一轮对话中某 Skill （如“事件查询”）输出了一个数据列表，系统必须将这些高阶结果的底层真实标识符（如 API 返回的隐藏 `uuid` 或 `rule_id`）暂存入该会话的上下文中。在后续轮次中，当用户下达诸如“查看序号3的事件详情”或“直接把第1个和第2个禁了”等自然指令时，系统能够自动匹配内存，将用户口中的“序号3”精准转换为前序保存的对应真实 `uuid`，从而再传递给下一个 Skill 触发执行。

### 2.4 安全事件管理 Skills

#### 2.4.1 核心要求与时间抽象
- 各类安全事件管理技能除了基础业务逻辑外，必须内置 **时间序列化工具**：LLM 提取的时间参数通常为自然语言（如“最近7天”），Skill 类层级需要将其转换为 API 所需的精确绝对时间戳（64位长整型）格式，确保“最近发生时间 (`startTimestamp`, `endTimestamp`)”的精准过滤。

#### 2.4.2 事件查询
- **需求描述**：多维度查询安全事件。
- **查询过滤参数（需对齐API规范）**：
  - **时间范围**：经转化后对应 `startTimestamp` 和 `endTimestamp`。
  - **事件等级**：严重(`4`)、高危(`3`)、中危(`2`)、低危(`1`)、信息(`0`)。
  - **处置状态**：待处置(`0`)、处置中(`10`)、已处置(`40`)、接受风险(`60`)等。
  - **模糊检索或属性过滤**：支持通过其它资产或平台分组 ID 检索。
- **状态联动约束**：查询成功后，必须将响应中的事件列表 `uuId` 数组按序存入当前对话上下文缓存 `SkillContextManager` 中，供后续技能映射使用。
- **结果展示**：以事件列表表格呈现、支持关键信息高亮（如展示 `uuId`, `name`, `incidentSeverity`, `dealStatus`, `hostIp` 等）。

#### 2.4.3 事件详情
- **需求描述**：查看单个事件的完整信息。
- **参数动态解析**：支持接收用户自然语言中如“第3个”、“刚刚那个”等基于序号的指示代词。Skill 内部负责从上下文中读取出对应的真实业务 `uuId`。
- **展示内容**：基本信息（包含 `gptResultDescription` 等 AI 研判结论）、攻击时间线、关联IP实体信息、详细举证证据。

#### 2.4.4 事件处置
- **需求描述**：更新事件处置状态。
- **同质联动**：支持批量操作，同样依赖于上下文缓存去解析诸如“把前两个标记为已处置”并转化为批量 `[uuId_1, uuId_2]`。
- **状态流转**：待处置 → 处置中 → 已处置，或标记为"已挂起/已忽略"。支持添加处置备注。


### 2.5 实体封禁管理 Skills

#### 2.5.1 封禁状态查询
- **需求描述**：查询实体（资产维度）在防火墙或端侧设备上的封禁状态。
- **核心查询参数 (匹配 api `searchInfos`)**：
  - 封禁资产 (`view` 字段，匹配具体的 IP / 域名 / URL)
  - 备注信息 (`reason`)、规则名称 (`name`)
  - 封禁状态过滤（如 `block success`, `block failed`, `unblocked` 等）
- **上下文联动**：查询完毕后，需将匹配到的 `规则ID (id)` 存入上下文，方便后续如“解封”类操作联动寻址。

#### 2.5.2 执行实体封禁 (新增网侧封禁)
- **需求描述**：在防火墙设备（AF）上创建封禁规则。
- **必填与枚举参数（需严格校验）**：
  - **实体类型 (`blockIpRule.type`)**：明确指定是 `SRC_IP` (源IP)、`DST_IP` (目的IP)、`URL` (链接) 或 `DNS` (域名)。
  - **封禁值 (`blockIpRule.view`)**：支持 IP 或 域名的数组。
  - **匹配模式 (`blockIpRule.mode`)**：固定或默认为 `in`。
  - **封禁时长 (`timeType`)**：必须指出是 `forever` (永久) 还是 `temporary` (临时)。如果是临时，还需提示或默认补充 `timeValue`（时长）和 `timeUnit`（单位，如 `d`/`h`/`m`）。
  - **设备信息 (`devices`)**：封禁必须指定联动的设备 (如 `devId`, `devName`, `devType: AF`)。可通过前置调用设备列表查询接口自动补充或让用户指定。
- **参数补齐机制**：当必须配置实体类型（IP/URL）、甚至设备或时长有缺失时，Skill 必须挂起当前执行流程，并将状态抛出请求大模型向用户发起追问确认。

#### 2.5.3 实体查询 Skill (第三方情报集成)
- **需求描述**：查询关联外网实体地理、威胁情报、历史攻击等信息。
- **凭证注入**：Skill 需要支持从全局系统配置/环境变量中动态读取（如配置的**微步在线 API Key**），未配置时引导用户去设置页面配置。

### 2.6 日志统计分析 Skills

#### 2.6.1 网络安全日志数量统计
- **需求描述**：提供指定时间段内（结合时间解析转化）日志聚合视图。
- **结果交付**：不仅仅交付文本统计数据，需要输出可供前端渲染的趋势图表数据结构（如 Echarts Option JSON），供 Web 前端多模态展示。


### 2.7 场景流编排引擎 (Workflow System)

#### 2.7.1 架构定义与区分
该模块不可与一般交互式 Skill 混为一谈。它是一个由事件触发（CRON 定时 / 手动立即触发）的 **自动化工作流（Workflow / DAG）引擎**。它本身是一条编排好的管道，串接了多个原子化的 Skill 进行静默执行。

#### 2.7.2 每日严重/高危事件闭环场景
- **用户可配置侧**：允许用户保存预没置（如触发周期、触发需覆盖的事件等级等）。
- **执行管线 (Pipeline Steps)**：
  1. 节点一：调用 `事件查询 Skill` 捞取当日配置等级的目标事件。
  2. 节点二：并行调用 `事件详情 Skill` + `实体查询 Skill` 自动化拉取上下文与情报。
  3. 节点三：大语言模型（LLM）推理归纳子节点，结合拉取的前置数据，生成自动化处置建议。
  4. 节点四：若需人机交互（Human-in-the-loop），则将汇总卡片推送到前端界面，等待用户人工 Check 确认。
  5. 节点五：确认后调用 `事件处置 / 封禁 Skill` 落地变更。
  6. 节点六：组装处理结果，触发外部 Webhook（如推送到企业微信机器人）。
