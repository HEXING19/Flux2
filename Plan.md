# Flux - 基于大语言模型的智能安全运营平台 (XDR) 开发方案规划

通过对 `PRD.md` 的深入研读，我已经充分理解了该项目的定位、核心功能以及特殊的设计约束。本项目不仅仅是一个传统的聊天机器人，而是一个**“意图驱动 + 环境感知 + 执行闭环”**的智能化企业安全运营系统。

以下是针对该项目所梳理的最详尽的架构规划与分阶段实施方案：

---

## 🧭 一、 核心架构设计与技术选型

为了满足“轻量化、易打包部署、面向对象(OOP)规范”的约束条件，系统整体架构设计如下：

### 1. 技术栈选型建议
* **后端引擎**: Python (FastAPI/Flask 等轻量框架)，适合处理 LLM 对接、并发请求、DAG 工作流以及轻便的 PyInstaller/Docker 一键打包。
* **前端交互界**: Vue3 / React (Tailwind CSS 等)，需要强调现代化、极具科技感的高逼格 UI 设计（特别是落地页）。
* **轻量级持久化**: SQLite 数据库（零配置，直接内嵌，完美切合“快速在客户侧部署验证”的需求）。

### 2. 核心模块边界
1. **多模态对话网关 (Conversation Gateway)**：负责接收用户自然语言，处理流式输出，内置 LLM 路由器适配（智谱/OpenAI/DeepSeek等）。
2. **能力执行引擎 (Skill Engine)**：基于纯 OOP 思想构建，负责所有原子化技能的挂载、参数序列化、以及**渐进式追问防线**的拦截与抛出。
3. **上下文寻址中心 (Context & Mapping Center)**：会话状态管理 (`SkillContextManager`)，处理“自然语言序号/代词”与“底层真实系统 ID/UUID”的映射转换。
4. **DAG 工作流引擎 (Workflow Engine)**：独立于即时对话系统外，负责由 CRON 或手工触发的多节点串行/并行调度，支持任务挂起（Human-in-the-loop 人工审批）。

---

## 🗺️ 二、 详细开发阶段拆解 (Phases)

整个项目开发将被严格拆分为 5 个核心里程碑（Milestones），以保证从地基到上层自动化逻辑的稳步推进。

### 阶段一：基础底座构建与“高逼格”落地页接入 (M1)
**重点**: 解决项目的入口问题、前端美观度以及基础连通性。

* **[1.1] 认证与落地页开发**:
  * 整合现有的 `py` 联动码登录逻辑代码。
  * 根据现代安全软件风格，设计并开发出炫酷、富有科技感的登录落地页（具有粒子动画、数据流转等微动效背景）。
* **[1.2] 多模型 LLM 连接池实现**:
  * 搭建各大基座模型（智谱、OpenAI、DeepSeek、自定义 Endpoint）的 API Client。
  * 实现 API 密钥的配置界面、全局存储配置、提供商连通性测试按钮以及失败自动重试机制。
* **[1.3] 前后端长连接通道搭建**:
  * 实现 WebSocket 或 SSE（Server-Sent Events），确保 AI 的回复具有“打字机”流式效果。

### 阶段二：会话状态记忆与上下文映射机制 (M2)
**重点**: 实现自然语义中指代词（“那个”、“第一条”）与真实底层 UUID 的桥接，此为本项目用户体验成败的核心。

* **[2.1] ContextManager (状态管理器) 开发**:
  * 开发 `SkillContextManager` 类，按会话 Session ID 隔离存储。
  * 维护**“参数与意图继承内存”**：当检测到当前对话的依赖参数（如 IP、事件）为空时，自动向上多轮历史追溯填充。
* **[2.2] 关联索引池 (Index Dictionary) 开发**:
  * 当任何查询类的 Skill 返回列表数据给用户前，在 ContextManager 中自动生成 `[序号前缀 / 指代符] <-> [业务UUID / 规则ID]` 的双向映射字典。
  * 在解析意图阶段，若发现“序号类”词汇，自动转换并覆写系统内部变量后再下发给被调用的 Skill。

### 阶段三：面向对象的 Skills 系统与安全 API 对接 (M3)
**重点**: 根据 `docs/api-ref` 落地各项原子技能，强调“渐进式追问防线”。

* **[3.1] Skill 抽象积木 (Base Class System)**:
  * 定义 `BaseSkill`：包含 `__init_schema__`（约束入参）、`execute_api_requester`（发送HTTP请求）、以及内置的时间解析钩子。
  * 实现**“时间序列化中间件”**：拦截如“最近三天”、“昨天”的文本，自动转化为底层所需绝对绝对时间戳（64位长整型 `startTimestamp` 与 `endTimestamp`）。
  * 实现**“渐进式追问防线” `MissingParameterException`**: 当提取出的参数不足以调用 API 时，主动 Stop 执行流程，向 LLM 返回特定结构，指挥其以自然语言询问用户。
* **[3.2] 安全事件管理 Skills 开发**:
  * `EventQuerySkill`: 覆盖等级、处置状态等条件，写入 uuid 到上下文，输出支持高亮渲染的表格。
  * `EventDetailSkill`: 支持上下文取 uuid，展示基本信息、时间线、IP与举证等。
  * `EventActionSkill`: 接收批量 uuid，状态流转控制(0->10->40/60)，备注记录。
* **[3.3] 实体封禁管理 Skills 开发**:
  * `BlockQuerySkill`: 查询封禁状态，提取 `rule_id` 入库。
  * `BlockActionSkill (网侧执行)`: 核心防控！拦截类型(`SRC_IP`/等)、枚举方向(`in`)、封禁时长(`forever`/`temporary`)等。**必须**结合设备列表接口强制防呆追问缺失参数。
  * `EntityQuerySkill`: 接入微步情报体系，支持从全局 Config 中动态读取其专属 API Key 凭证。
* **[3.4] 日志统计 Skill 开发**:
  * `LogStatsSkill`: 获取聚合数据，不仅输出描述文本，**核心是要返回 Echarts Option JSON 协议的结构** 给前端，用于多模态动态图表渲染。

### 阶段四：场景流编排引擎 (Workflow System) 实现 (M4)
**重点**: 把独立执行的原子化 Skill 串成静默/自动化流水线，实现真正的 SecOps 闭环。

* **[4.1] DAG 执行控制器开发**:
  * 不混淆于交互场景，提供任务管道（Pipeline）声明抽象。支持 Cron 调度触发引擎。
* **[4.2] 【每日严重高危闭环】主流程落地**:
  * **节点 1**: 定时拉取高危告警。
  * **节点 2 (异步并发)**: `EventDetailSkill` 与 `EntityQuerySkill` 同步深挖告警详情与外部实体恶意情报。
  * **节点 3 (LLM Summary)**: 大模型组装汇总前两个节点的证据节点链，输出专家研判意见与推荐处置结论。
  * **节点 4 (人工审批 Human-in-the-loop)**: 将流程状态标记为强挂起 (`Suspended`)，推送到前端界面通知队列中等待人工 Click 审核。
  * **节点 5 & 6**: 审批下发后自动触发 `BlockActionSkill` 或状态调整，并调用 Webhook (企业微信/钉钉) 广播战果信息。

### 阶段五：一键交付与体验调优 (M5)
**重点**: 满足项目最终在客户侧的无门槛部署验证诉求。

* **[5.1] 一键打包部署**:
  * 利用 PyInstaller 打包 Python 后端以及已构建好的前端静态资产（或提供极其精简的离线 Docker-compose 包），实现解压后“双击即运行 / 一句指令即拉起”的能力。
* **[5.2] 拟真对抗与压力自测**:
  * 模拟各路径缺省场景：刻意少说一个 IP 或时效，测试 AI 是否精准触发反问防线。
  * 列表指针抗错能力：测试各种边缘指示代词（如“跳过第一个，把剩下全封了”），打磨 `SkillContextManager` 的鲁棒性。

## 💡 三、 难点与成功关键评估

1. **大模型的“过度脑补”(Hallucination) 管理**：
   * **方案**：对于具有不可逆破坏性的操作（如封禁 Skills），系统不能盲目听从模型的自行伪造。所有的 `enum` (类型) 与 `timeType` 必须在 Python 层面再次通过 `Pydantic` 或正则约束，只要非法立刻打回给大模型要求其重组用户的问句。
2. **多模态与复合结果输出的界面兼容**：
   * **方案**：采用"卡片化插槽(Slot)"理念设计前端聊天流界面。后端在回复时不仅仅是 Markdown，更通过 Payload 夹带不同类型标记（`type: text | table | echarts_graph | approval_card`），前端组件化动态拼解不同形态的安全信息。
3. **安全操作二次阻断**：
   * **方案**：针对 `EventActionSkill` 与 `BlockActionSkill` 动作必须强置状态位，前端界面拦截执行请求并自动弹窗红色阻拦二次确认组件。

我们将以此计划作为基准线路图进行敏捷迭代展开。
